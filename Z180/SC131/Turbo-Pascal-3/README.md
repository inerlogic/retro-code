# Game of Life screensaver (Turbo Pascal 3, color)

**MILLIFE.PAS** is a Pascal port of the Aztec-C `MILLIFE.C`/`MILLIF2.C` 1D cellular automaton (see [`../../../Aztec-C/life/`](../../../Aztec-C/life/)) — same 4-neighbor-sum rule and loop detection (an 8-generation checksum history that triggers a re-seed if a checksum repeats), but each live cell also carries a color (an ANSI background color code 41–47) that blends from its neighbors' colors as generations evolve, instead of the plain `*`/space display of the C version. Uses the same LFSR-based `GetRnd` as the C original for its initial seeding.

Named in tribute to **Jon Millen**, author of "One-Dimensional Life" in BYTE Magazine, Vol 3 No 12, December 1978 — the five-cell YYXYY neighborhood rule this program implements. See [jonmillen.com/1dlife](https://jonmillen.com/1dlife).

## HEXLIFE.PAS

A different kind of Life screensaver, born out of watching `AUTOLIF.BAS` (see [`../../BASIC-80/`](../../BASIC-80/)) settle permanently at generation 788 (seed 23432) and just sit there forever afterward, once its identical two phases were confirmed byte-for-byte via diff. This version exists specifically so that doesn't happen.

- **Classic 2D Conway's Life** (8-neighbor sum, same rule as `AUTOLIF.BAS`/`LIFE.BAS`), not Millen's 1D rule — what's borrowed from the 1D life programs is the *technique*, not the rule.
- **Fully toroidal** — wraps all four sides (a "ball" rather than a cylinder or a bounded dead-border grid like `AUTOLIF.BAS` uses).
- **Displayed as hex digits in a fixed-position bordered playing field**, redrawn in place each generation rather than scrolling — each hex digit packs 4 consecutive cells (same MSB-first packing scheme as the hex-encoding change made to `AUTOLIF.BAS`, and the IMSAI Z80/8080 assembly programs elsewhere in this repo).
- **Silent auto-reseed** using the same rolling-checksum technique as `MILLIF2.C` (`checksum := checksum*3 + cell`, compared against a short history) — but silent, no announcement, matching `MILLIFE.C`'s quieter sibling rather than `MILLIF2.C`'s visible one.
- **Grid deliberately smaller than the full terminal:** 40 hex columns × 18 rows (160×18 cells) rather than filling all 80×24. Two full-terminal-width grids would have needed 15,360 bytes — a meaningful fraction of a modest CP/M TPA (`54.0K TPA` seen in this repo's own boot logs). At 160×18 it's 5,760 bytes, a much more comfortable margin. The border and "GENERATION:" label are static chrome drawn once at startup/reseed (never redrawn during normal per-generation steps, since they don't change) — and can be turned off entirely via the `SHOW_CHROME` constant for the borderless look, given modern/virtual terminals don't have the burn-in concern a border traditionally exists to help with.

**Status: confirmed compiling and running on real hardware** — border/label render correctly, hex digits fill the field as expected. One real bug found and fixed along the way: the two string-size type declarations (`THexDigits`, `TNumStr`) were mistakenly placed in the `const` block instead of `type` — Pascal's `const` section can only hold literal values, not type definitions, and that's exactly the kind of thing that throws a `';' expected` compile error. Moved them to `type` where `TGrid` already lives, fixed.

**Pacing:** initially, no artificial delay was needed for `HEXLIFE.PAS` — the natural computational cost (2,880 cells × 8-neighbor lookups, each going through function calls rather than inlined math) ran noticeably slow rather than fast. That changed once color was added to `HEXLIFEC.PAS` — see the computation optimization below, applied to both files.

**Optimized: `Step`'s neighbor-sum computation, applied to both `HEXLIFE.PAS` and `HEXLIFEC.PAS`.** The original called out to separate `NeighborSum`/`WrapCoord` functions ~49,000 times per generation (2,880 cells × ~17 calls each: one `NeighborSum` call plus 16 `WrapCoord` calls) — and beyond the function-call overhead itself, `WrapCoord(X-1, GRIDW)` was being recomputed three separate times per cell (once for each of the three DY pairings that use it), purely redundant work. Replaced with inlined math that hoists the X-wrap calculation outside the inner Y loop entirely — since it only depends on X, not Y, it's now computed once per column (`GRIDW` times total) instead of once per cell (`GRIDW*GRIDH` times). Verified in Python across all 2,880 cells (edges included) that the optimized math produces results identical to the original before making this change — zero mismatches. Confirmed on real hardware: `HEXLIFE.PAS` runs at roughly one generation per second.

**Header expanded to `Seed: NNNNNN  Generation: NNNNNN  Checksum: NNNNNN`.** Switched from Pascal's built-in `Random()`/`Randomize()` (which seeds from an internal clock, giving no visible/enterable number) to the same seedable LFSR `MILLIFE.PAS` already uses (`RandSeed`/`GetRnd`), so a real seed number can be typed in (matching `AUTOLIF.BAS`'s seed-prompt convention) and displayed. Since `GetRnd` only returns a single bit, `GetRndPct` builds a 7-bit value from 7 calls and thresholds it to approximate the ~35% alive-density `AUTOLIF.BAS` uses. The checksum (previously computed silently and discarded each generation) is now cached and displayed too, so the loop-detection mechanism is actually visible instead of a black box — though the *reseed* itself is still silent, no announcement, matching the original design.

**Resolved:** the color-variant question got settled (see `HEXLIFEC.PAS` below).

**Fixed: ESC didn't reliably quit the program.** Checking `KeyPressed` only once per generation (~2 seconds of uninterrupted computation with essentially zero I/O calls in between) reliably failed to catch ESC — confirmed via isolated diagnostic stubs, even a single clean press with no spamming, no display/output logic involved at all. Ruled out along the way: `KeepRunning` being silently reset elsewhere (checked, wasn't happening), and console output volume interfering with input detection (batched `DrawGrid`'s 720 individual character writes into one string-write per row — genuine perf win on its own, but didn't fix ESC). The actual fix: check `KeyPressed` from *inside* `Step`'s loop, roughly every 200 cells (~14 times per generation) instead of once between generations — confirmed working via a matching isolated test. The exact underlying mechanism isn't fully understood (possibly some UART/BDOS-level behavior on this specific machine that needs reasonably frequent polling to catch an incoming byte before it's lost), but the fix itself is confirmed on real hardware, independent of the mystery around *why* it works.

## HEXLIFEC.PAS

Color variant of `HEXLIFE.PAS` — everything the same (toroidal grid, seedable LFSR, silent checksum-based reseed, the confirmed ESC fix) except `DrawGrid`, which colors each hex digit's *text* by its value (0 ascending) instead of printing it plain.

- **Color table:** a genuine brightness-ordered heat-map gradient — 0 = dark gray through F = full white, monotonically increasing — rather than a direct conversion of `HEXPI.PAS`'s table (that approach grouped colors by *family* — all blues, then all greens, etc. — not by perceived brightness, so it wasn't actually a gradient in the sense wanted here). Only 15 genuinely distinct, visible ANSI foreground codes exist in this basic 16-color palette once true black is excluded, so one repeat (bright blue, used twice) was unavoidable — placed at the gentle blue-to-cyan transition rather than disrupting the E/F "near blazing to full white" pair that was specifically the point.
- **Value 0 is dark gray, not invisible black** — an explicit, deliberate choice (unlike the first draft, which used true black as the "honest direct equivalent" of `HEXPI.PAS`'s own mapping — that version's gone now, this is the real one).

**Fixed: the reset-per-cell fix made the whole redraw visibly slow — a "wave" writing top to bottom instead of an instant redraw, confirmed on hardware.** Doubling the escape codes sent per cell (720 cells × 2 codes each, every generation) was the direct cause. Fixed with change-detection: color codes are now only emitted when a cell's value actually *differs* from the previous cell's — if neighboring cells share a value (common in Life grids, where the neighbor-counting rule tends to create locally coherent regions), the character alone is written and the already-active color is reused. Doesn't change the correctness of the bold-bleed fix at all (still resets whenever color needs to genuinely change) — just skips the redundant work when it doesn't.

**Still not fast enough — collapsed from 16 colors to 8.** Even after the above, the wave persisted on hardware, and separately, the bold/plain distinction the 16-color gradient depended on turned out unreliable on this terminal (confirmed: values 1/2, 4/5, 6/7, C/D, E/F looked identical even after the reset-per-cell fix made the *code* correctly distinct — likely a rendering-level limitation rather than anything fixable in software). Collapsed to 8 genuinely distinct base hues (`Bucket := Nibble div 2`) sidesteps that ambiguity entirely, and — more importantly for the wave — directly reduces the number of transmitted color codes under change-detection: two adjacent cells that previously needed different bold/plain codes (e.g. nibble 4 and 5) now share a bucket and emit nothing extra between them. A real fix aimed at the actual bottleneck, not just a simplification for its own sake. Confirmed on hardware: the wave changed character from chunky top/bottom-half jumps to a smooth continuous flow, correlating visibly with grid density (more dead cells = less wave) — direct evidence the optimization is working as intended, not just plausible in theory.

**Further optimized: eliminated the separate reset entirely, by making the palette consistently bold.** A sharp catch, worth recording precisely: the reset was never needed because bold *changes* between cells — it was needed because the old palette *mixed* bold and plain codes inconsistently. If every one of the 8 bucket colors includes the bold flag, every color-change code re-asserts "bold on" as part of itself; there's never a moment where bold needs *clearing*, only *reasserting*, and reasserting doesn't need a reset first. Changed all 8 bucket colors to their bold variants (previously buckets 1-6 were plain, only 0 and 7 were bold) and dropped the separate `\x1b[0m` prefix from the per-cell color codes — roughly halving the escape sequence length for every transition. The end-of-`DrawGrid` reset (preventing the last cell's color from bleeding into the plain header text) is a different job and is unchanged.

**Status: confirmed working on real hardware, including multiple confirmed oscillators (see below).** Based directly on `HEXLIFE.PAS`'s already-hardware-confirmed file (same ESC fix, same seed/checksum header, same toroidal math), with only `DrawGrid` changed. One real bug caught and fixed before export: `Copy()` returns a string, not a `Char`, so the initial `DigitChar: Char` declaration was a type mismatch — corrected to `DigitChar: string[1]` before this ever reached a compiler.

**Added: `HIDE_CURSOR` toggle**, same pattern as `SHOW_CHROME` — deliberately left as a compile-time flag rather than picking one and committing, since the cursor tracking where the redraw currently is could reasonably be read as either an annoying visual artifact of the wave, or useful feedback for actually seeing it. Uses `HEXPI.PAS`'s already-proven DECTCEM escape codes (`\x1b[?25l` hide, `\x1b[?25h` show). Restoring the cursor on exit is unconditional, regardless of the toggle — harmless if it was never hidden, essential if it was, so the terminal is never left in a bad state either way. Confirmed on hardware: hidden cursor preferred, "very clean" look.

### Confirmed oscillators (seed 23432, toroidal grid)

Caught on video and cross-checked by seeing the same transitions independently at multiple grid locations — genuine local oscillators under standard 8-neighbor Conway's Life, not artifacts of the hex display:

- Three `8`s in a column: top and bottom flip to `0`, center flips to `C` (with a `1` appearing to its left)
- Three `2`s in a column flip to `7`
- A column of `696` — stable, no change
- Three `1`s in a column flip to `3` (with an `8` appearing to its right) — same transition confirmed at two separate, independent grid locations
- Two `3`s stacked vertically — stable, no change
- Three `4`s in a column, offset by one column, flip toward `E`, with `C`s appearing above/below the top and bottom `4`s — unclear whether the `C`s contributed to stability, since this one got overrun by incoming activity from elsewhere after roughly a dozen cycles rather than persisting indefinitely like the others

**Fixed: displayed seed now updates on reseed, in both `HEXLIFE.PAS` and `HEXLIFEC.PAS`.** Previously, the *displayed* `Seed: NNNNN` stayed frozen at the original startup value across a silent reseed, even though the grid itself genuinely changed (`RandSeed`'s live LFSR state was never reset to match, so the new soup was real, just not reflected in the header number). On reseed, `SeedVal` is now rotated and incremented, then `RandSeed` is reset to match it — so the header number changes to a new value every time a reseed fires, and the displayed seed and the LFSR's actual internal state stay in sync going forward. Confirmed working on real hardware, both files.

## HEXLIF16.PAS

The 16-color version of `HEXLIFEC.PAS`, kept around under its own name rather than deleted once the 8-color collapse replaced it — genuinely too slow on this hardware (the visible "wave" that motivated the 8-color version in the first place), but worth having on record as a concrete before/after speed comparison, and because a 16-step gradient isn't inherently a bad idea — it was specifically *this* machine's serial transmission speed and terminal rendering quirks that made it impractical here.

*Note to self: if development for MS-DOS on a 386 ever happens, this is worth revisiting.* A 386 running DOS natively (rather than over a slow serial link into a Z180 CP/M box) would sidestep the actual bottleneck entirely — direct video memory writes instead of ANSI escape sequences over serial, and no equivalent transmission-speed ceiling — so the 16-color gradient might work perfectly well there with zero changes needed. Just a flag for future-me.
